## 9. 자동화 제안

반복 3회 이상 패턴에서 도출된 자동화 후보:

| # | 반복 패턴 | 횟수 | 제안 | 우선도 |
|:-:|----------|:----:|------|:------:|
| 1 | RAG 채널 데이터 수집→벡터DB 세팅 | 3+ | `/rag-channel <채널명>` 커맨드 신설 | **높음** |
| 2 | HTML 캡쳐→문서 삽입 | 4+ | `--capture` 옵션으로 캡쳐+삽입 원스텝화 | **높음** |
| 3 | Docker 재빌드/실행 | 6+ | `/docker rebuild` 또는 Hook 자동화 | **높음** |
| 4 | Annotation 이미지 분석 | 4+ | `--anno` 옵션 개선 (배치 분석 지원) | 중간 |
| 5 | Markdown 이미지 링크 복구 | 3+ | post-edit Hook으로 링크 유효성 자동 검증 | 중간 |
| 6 | Confluence 인증 확인 | 3+ | `--con` 실행 전 인증 자동 체크 추가 | 낮음 |
| 7 | 네트워크 접근 설정 | 3+ | 프로젝트 템플릿에 네트워크 설정 포함 | 낮음 |

## 10. 종합 인사이트

```
  +------------------------------------------+
  |           반복 패턴 분류                  |
  +------------------------------------------+
  |                                          |
  |  [A] 동일 작업 반복 (자동화 가능)        |
  |      - RAG 세팅 (3회)                    |
  |      - Docker 재빌드 (6회)               |
  |      - 캡쳐+삽입 (4회)                   |
  |                                          |
  |  [B] 재작업 (품질 이슈)                  |
  |      - "다시" 15회, "수정" 17회          |
  |      - 이미지 링크 깨짐 반복             |
  |      - 설계 재작업 반복                  |
  |                                          |
  |  [C] 확인/질문 반복 (정보 갭)            |
  |      - "맞아?" "했어?" 패턴              |
  |      - Confluence 인증 상태 반복 확인    |
  |      - Docker 상태 반복 확인             |
  |                                          |
  +------------------------------------------+
```

### 핵심 발견

1. **자동화 ROI 최고**: Docker 관련 (6회) + 캡쳐/삽입 (4회) + RAG 세팅 (3회) = **13건의 반복을 커맨드/Hook으로 제거 가능**
2. **재작업 비율 높음**: "다시"+"수정" 32회는 1회차 품질이 기대에 못 미침을 시사. QA 사이클 강화 또는 Architect Gate 조기 개입 검토 필요
3. **`/auto` 과집중**: 전체 커맨드의 75.6%가 `/auto`. 다른 커맨드(`/check`, `/debug`, `/research`) 활용도가 낮아 `/auto` 내부에서 모든 작업을 처리하는 경향
