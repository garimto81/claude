# Context Specialization Strategy 검증 최종 보고서

**검증 날짜**: 2026-02-25
**검증 방식**: 적대적 (Adversarial) HEAVY PDCA
**검증 완료**: Phase 1(플랜) → Phase 3(7개 실증 태스크) → Phase 3.2(Architect 독립 검증)

---

## 1. Executive Summary

### 검증 목적

Context Specialization Strategy 문서는 논문 arxiv:2602.11988의 발견을 근거로 Claude Code 워크플로우의 컨텍스트 최적화 방안을 4개 전략(A~D)으로 제안했다. 본 보고서는 각 전략이 실제 Claude Code SDK 제약 하에서 구현 가능한지, 안전 규칙을 보존하는지, 유지보수 비용이 합리적인지를 실증 검증한 결과를 종합한다.

### 핵심 판정 요약

7개 실증 태스크와 Architect 독립 검증 수행 결과:

| 전략 | 판정 | 핵심 사유 |
|------|------|-----------|
| A (CLAUDE.md 슬림화) | **조건부 수용** | 프로필 분기는 기각. 단일 CLAUDE.md 정적 축소만 유효 |
| B (마이크로 스킬) | **기각** | 안전 규칙 우회 + 유지보수 5배 증가 |
| C (동적 프로파일링) | **기각** | session_init.py가 context 로딩을 제어 불가 — 순수 가상 설계 |
| D (규칙 태그) | **기각** | 태그 기반 선택 로드 API가 Claude Code에 존재하지 않음 |

### 핵심 수치

- **현재 항상 로드 실제 총계**: 857줄 (CLAUDE.md x2: 336줄 + rules x6: 521줄)
- **전략 문서 추정**: 706줄 — 21% **과소평가**
- **달성 가능한 최대 감소**: Phase 1 슬림화 후 681줄 (20.5% 감소)
- **전략 목표 -50%**: **달성 불가** — rules 521줄이 항상 로드되어 물리적 하한 존재
- **확정된 결함**: 12개 중 10개 확정, 2개 추가 확인 필요

### 즉시 실행 가능한 경로

SDK 변경 없이 4개 대안(정적 슬림화, rules 물리적 분리, SKILL.md 참조 분리, hook 컨텍스트 요약 주입)을 조합하면 최대 ~176줄(20.5%) 감소를 안전하게 달성할 수 있다. 전략 목표의 절반 수준이나, 안전성을 보장하면서 즉시 실행 가능한 현실적 경로다.

## 2. 검증 대상 전략 개요

**전략 A: 프로젝트 타입별 CLAUDE.md 슬림화**
6개 도메인 프로필(minimal, web, data, media, ops, docs)을 `~/.claude/profiles/`에 작성하고, 각 서브 프로젝트가 해당 도메인의 최적화된 CLAUDE.md를 로드하도록 세션 시작 시 전환한다. 모든 프로젝트에 공통으로 적용되는 불필요한 규칙과 중복 정보를 제거하여 컨텍스트 부하를 706줄에서 350줄로 줄이는 것을 목표로 한다.

**전략 B: 도메인별 마이크로 스킬**
`/auto`의 5-Phase PDCA를 모든 작업에 강제하는 현행 구조 대신, 작업 규모와 도메인에 따라 분화된 스킬(`/quick`, `/fix`, `/auto-media`, `/auto-data`, `/auto-ops`)을 도입한다. 특히 `/quick`은 10줄 이하 소규모 변경에 PRD-First와 Architect 검증을 생략하여 오버헤드를 5분에서 30초로 단축한다.

**전략 C: 동적 컨텍스트 프로파일링**
`session_init.py`에 프로젝트 타입 자동 감지 로직(`detect_project_type()`)을 추가하고, 감지 결과에 따라 최적 컨텍스트를 로드(`load_context()`)하는 자동화 파이프라인을 구축한다. 프로젝트별 `profile.yaml`이 사용할 에이전트 목록, 활성/비활성 규칙, 컨텍스트 예산을 선언적으로 정의한다.

**전략 D: 규칙 선택적 활성화**
현재 6개 규칙 파일을 항상 전부 로드하는 구조 대신, `[ALL]`, `[DEV]`, `[MEDIA]`, `[DOCS]` 태그로 규칙을 분류하여 프로젝트 타입에 해당하는 태그의 규칙만 로드한다. MINIMAL 프로젝트는 `[ALL]` 태그 규칙만 적용하여 521줄에서 30줄 수준으로 감소시키는 것을 목표로 한다.

## 3. 결함 판정 매트릭스

### 3.1 CRITICAL 결함 (전략 무효화 수준)

#### F-1: CLAUDE.md 런타임 교체 불가
**판정**: 확정

**핵심 증거**:
- `C:\claude\.claude\settings.json` 실증 확인: `contextFiles`, `profileSwitch`, `active_rules` 필드가 존재하지 않는다.
- `session_init.py` 반환 스키마 실증 확인: hook은 `{"continue": true, "message": "..."}` 형태만 반환 가능하며, context 로딩 경로를 변경하는 필드가 없다.
- Claude Code 공식 동작: CLAUDE.md는 세션 시작 시 정적 로드이며, `~/.claude/profiles/media.md` 같은 파일명은 자동 로드 대상이 아니다.

**영향 범위**: 전략 A의 프로필 분기 방식 무효화. 전략 C의 동적 프로파일링 핵심 메커니즘 무효화. 전략 A/C에서 제안한 `load_context()` 함수가 구현되더라도 실제 context 로딩을 변경할 수 없다.

---

#### F-2: rules 파일 줄 수 41% 과소평가
**판정**: 확정

**핵심 증거**:
- 실측 값: `04-tdd.md`(18줄) + `08-skill-routing.md`(53줄) + `10-image-analysis.md`(75줄) + `11-ascii-diagram.md`(141줄) + `12-large-document.md`(98줄) + `13-requirements-prd.md`(136줄) = **521줄**
- 전략 문서 추정값: ~370줄 (41% 과소평가)
- 결과: 전체 항상 로드 총계가 전략 추정 706줄이 아닌 실제 **857줄**임

**영향 범위**: 전략의 컨텍스트 감소 목표 수치 전체가 부정확한 기준값에서 산출된 것이다. rules 파일 521줄은 CLAUDE.md 최소화만으로는 변경되지 않는다.

---

#### F-3: 전략 감소 목표 (-50%) 달성 불가
**판정**: 확정

**핵심 증거**:
```
현재 항상 로드 실제 총계: 171 + 165 + 521 = 857줄
Phase 1 후 최선 시나리오: 80 (global) + 80 (project) + 521 (rules 변경 없음) = 681줄
실제 달성 가능 감소: 857 - 681 = 176줄 (20.5%)
전략 목표: -50% (353줄 감소 필요) → 달성 불가
```

**영향 범위**: 두 CLAUDE.md를 완전히 빈 파일로 만들어도 rules 521줄은 항상 로드되므로, 전략이 제시한 "706줄 → 350줄" 목표는 수학적으로 달성 불가능하다.

### 3.2 SIGNIFICANT 결함 (심각한 설계 결함)

#### S-1: 동적 프로파일링 구현 불가
**판정**: 확정

**핵심 증거**: `session_init.py`가 실제로 수행하는 작업(stale 정리, git 상태 읽기, TODO 로드, Junction 설정)은 모두 부수 작업이며, `{"continue": true}` 신호 반환이 전부다. SDK가 hook 반환값을 통해 context 로딩 경로를 변경하는 메커니즘을 제공하지 않는다.

**영향 범위**: 전략 C 전체가 현재 SDK 내에서 구현 불가능한 가상 설계다.

---

#### S-2: 마이크로 스킬 수동 동기화 부담
**판정**: 확정

**핵심 증거**:
- 제안된 5개 마이크로 스킬(`/quick`, `/auto-media`, `/auto-data`, `/auto-ops`, `/auto-web`)이 현재 모두 존재하지 않는다.

**영향 범위**: 보안 패치 전파 비용 5배 증가. fork divergence로 인한 장기적 보안 drift 확실.

## 4. 실행 가능한 대안

CRITICAL 결함 3건(F-1, F-2, F-3)이 전략 A/C/D의 핵심 전제를 무효화하므로, Claude Code의 실제 동작 메커니즘 내에서 실현 가능한 4개 대안을 제시한다.

### 대안 1: CLAUDE.md 정적 슬림화 (즉시 실행 가능)

전략 A의 프로필 분기 방식 대신, 단일 CLAUDE.md를 직접 축소한다.

**현재 구조**:
- `~/.claude/CLAUDE.md` 165줄 — 글로벌 지침
- `C:\claude\CLAUDE.md` 171줄 — 프로젝트 지침

**슬림화 원칙**: 에이전트가 없어도 파일/코드 탐색으로 알 수 있는 정보는 제거한다 (논문 황금 규칙).

**제거 대상 후보**:
- 아키텍처 설명 (코드 탐색으로 확인 가능)
- 에이전트 목록 (`.claude/agents/` 직접 로드됨)
- 참조 문서 목록 (파일로 존재하므로 중복)
- 중복된 핵심 원칙 (양 파일에 동일하게 존재하는 항목)

**보존 필수 항목**:
- API 키 금지 선언 (hook으로 100% 차단 불가)
- `TeamCreate → Task → SendMessage → TeamDelete` Agent Teams 프로토콜
- 절대 경로 원칙 (런타임 동작에 직접 영향)

**예상 효과**: 두 CLAUDE.md를 각 80줄로 축소 → 336줄에서 160줄 → 176줄 절감.

**기술 근거**: CLAUDE.md 파일 자체 편집은 SDK 제약 없이 항상 가능하다. 파일 분기(런타임 교체)가 아닌 내용 축소이므로 F-1 결함과 무관하다.

### 대안 2: .claude/rules/ 물리적 분리 (Junction 기반)

태그 시스템 대신, 서브 프로젝트별로 필요한 rules 파일만 Junction으로 연결한다.

**구현 방식**:
```
C:\claude\.claude\rules\          — 공통 규칙 (04-tdd.md, branch 관련)
C:\claude\.claude\rules-media\    — 미디어 전용 규칙 (10-image-analysis.md)
C:\claude\.claude\rules-dev\      — 개발 전용 규칙

서브 프로젝트 적용 예시:
  wsoptv_ott\.claude\rules\ → Junction → rules\ + rules-media\
  investment\.claude\rules\ → Junction → rules\ + rules-dev\
```

**기술 근거**: `.claude/rules/` 디렉토리의 파일을 물리적으로 분리하면 해당 디렉토리의 `.md` 파일만 로드된다. Junction은 `session_init.py`의 `setup_commands_junction()` 함수에서 이미 검증된 패턴이다.

**주의 사항**: 서브 프로젝트별 Junction 유지보수 부담이 있다. `session_init.py`에 프로젝트 타입 감지 후 Junction 자동 생성 로직을 추가하면 일회성 설정으로 해결 가능하다.

**예상 효과**: 미디어 프로젝트에서 개발 전용 규칙 로드 제거, 개발 프로젝트에서 OCR 규칙 로드 제거 → 세부 절감 폭은 프로젝트별 상이.

---

### 대안 3: /auto SKILL.md 참조 분리 (즉시 실행 가능)

마이크로 스킬 분기 없이, `/auto` SKILL.md에서 상세 내용을 REFERENCE.md로 분리한다.

**현재 구조**: `/auto/SKILL.md` 426줄 단일 파일

**분리 방식**:
```
/auto/SKILL.md       — 100줄 이하 (핵심 흐름 + 에이전트 호출 규칙만)
/auto/REFERENCE.md   — 326줄 (Phase별 상세, 복잡도 분기, 예외 케이스 등)
```

에이전트가 필요 시 `Read("REFERENCE.md")`로 참조한다.

**기술 근거**: SKILL.md는 스킬 호출 시 context에 로드된다. 파일 크기를 줄이면 직접적으로 context 절감 효과가 있다. 스킬 시스템에 상속 메커니즘은 없지만, 단일 파일 크기를 줄이는 것은 가능하다.

**보안 패치 보존**: REFERENCE.md 분리는 단일 파일 편집 원칙을 유지하므로, 마이크로 스킬 분기(S-2)와 달리 보안 패치가 1개 파일 수정으로 전파된다.

**예상 효과**: SKILL.md를 100줄로 축소 시 326줄 절감. REFERENCE.md는 필요 시에만 로드하므로 항상 로드 총계가 감소한다.

---

### 대안 4: Hook 컨텍스트 요약 주입 (안전성 강화)

CLAUDE.md 축소와 병행하여, 안전 규칙을 hook에 집중시킨다.

**구현 방식**:

`session_init.py`에 프로젝트 타입별 규칙 요약 메시지 주입:
```python
# 기존: {"continue": True, "message": "session info..."}
# 강화: context 교체가 아닌 핵심 규칙 보충 메시지 주입
message = f"""
[현재 프로젝트: {project_type}]
필수 규칙: API 키 금지 (Browser OAuth만), 절대 경로 사용, PRD-First
"""
```

`tool_validator.py`에 API 키 패턴 차단 추가:
```python
API_KEY_PATTERNS = ["sk-", "AKIA", "AIza", "Bearer "]
# Write/Edit 시 파일 내용에 이 패턴이 존재하면 차단
```

**기술 근거**: hook의 메시지 주입은 context 내용에 추가되므로, 슬림화된 CLAUDE.md를 보완하는 역할을 한다. context 교체(F-1 불가)가 아닌 context 보충이므로 SDK 제약 없이 구현 가능하다.

**예상 효과**: CLAUDE.md에서 API 키 금지 선언을 제거하더라도, hook 메시지로 매 세션 주입. tool_validator.py 패턴 차단으로 하드코딩을 추가 차단.

## 5. 우선순위 권장 사항

검증 결과를 기반으로 즉시 실행 가능 여부, 안전성, 효과 크기를 기준으로 우선순위를 정렬한다.

### P-1: 즉시 실행 — /auto SKILL.md 참조 분리 (대안 3)

**우선순위**: 최고
**이유**: SDK 제약 없음, 보안 패치 단일 파일 유지, 즉각적인 context 절감 효과.

| 항목 | 내용 |
|------|------|
| 실행 방법 | `/auto/SKILL.md`에서 Phase 상세/복잡도 분기/예외 케이스를 `REFERENCE.md`로 분리 |
| 예상 절감 | 항상 로드 326줄 감소 (SKILL.md 426 → 100줄) |
| 위험 | `Read("REFERENCE.md")` 호출 누락 시 상세 지침 누락 위험 — 핵심 흐름은 SKILL.md에 보존 |
| 검증 방법 | /auto 실행 후 Agent Teams 프로토콜이 정상 동작하는지 확인 |

---

### P-2: 즉시 실행 — CLAUDE.md 정적 슬림화 (대안 1)

**우선순위**: 높음
**이유**: 중복 정보 제거는 논문 근거 있음(+2.7% 효과). SDK 제약 없이 즉시 실행 가능.

**실행 순서**:
1. `~/.claude/CLAUDE.md`와 `C:\claude\CLAUDE.md`에서 중복 섹션 식별
2. 양 파일에서 동일하게 존재하는 내용 → 한 파일에만 유지
3. 코드 탐색으로 확인 가능한 아키텍처 설명 → 제거 또는 참조 링크로 대체
4. API 키 금지 선언 — **절대 제거 금지** (보존 필수)
5. Agent Teams 프로토콜 — **절대 제거 금지** (시스템 기능 정의)

**예상 절감**: 336줄(CLAUDE.md 합계)에서 160줄로 축소 → 176줄 절감.

---

### P-3: 조건부 실행 — Hook 안전성 강화 (대안 4)

**우선순위**: 보통
**이유**: CLAUDE.md 슬림화와 병행 시 필수. 단독 실행 시에도 안전성 강화 효과.

**조건**: CLAUDE.md 슬림화(P-2) 실행 전 또는 병행하여 적용. `tool_validator.py`에 API 키 패턴 차단 추가.

**주의**: hook 패턴 매칭은 100% 차단을 보장하지 않으므로, CLAUDE.md의 API 키 금지 선언을 완전히 대체하는 것은 권장하지 않는다.

---

### P-4: 장기 검토 — rules 물리적 분리 (대안 2)

**우선순위**: 낮음 (장기)
**이유**: 설계는 타당하나 Junction 유지보수 비용이 있다. session_init.py 자동화 없이는 수동 관리 부담.

**전제 조건**:
- rules 파일의 SDK 자동 로드 동작 공식 문서 확인 (M-3 해소 후)
- session_init.py에 프로젝트 타입 기반 Junction 자동 생성 로직 구현

**보류 이유**: M-3(rules 자동 로드 가정)이 미확인 상태이므로, 이를 확인한 후 설계를 확정한다.
