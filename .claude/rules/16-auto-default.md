# /auto 자동 실행 규칙 — 기본 워크플로우 강제

**트리거**: 사용자가 메시지를 보내는 모든 상황 (Phase 0 전 판단)

## 핵심 원칙

`/auto`는 기본 워크플로우다. 사용자가 `/auto`를 입력하지 않아도 작업 요청이면 자동으로 `/auto` 워크플로우를 실행한다.

## 판단 흐름

```
사용자 메시지 수신
  ├─ 예외 조건 해당? ──→ YES ──→ 해당 처리 (직접 응답, 다른 스킬 등)
  └─ NO ──→ Skill("auto", args="{사용자 요청 원문}") 자동 호출
```

## 자동 실행 대상 (YES → /auto)

| 패턴 | 예시 |
|------|------|
| 기능 구현/추가 | "로그인 기능 만들어줘", "API 추가" |
| 버그 수정 | "이 에러 고쳐줘", "동작 안 해" |
| 리팩토링 | "이 코드 정리해줘", "구조 개선" |
| 분석/리서치 | "이 코드 분석해줘", "대안 조사" |
| 설정/워크플로우 변경 | "hook 추가해줘", "설정 바꿔줘" |
| 문서 생성/수정 | "PRD 작성", "README 업데이트" |
| 테스트 작성 | "테스트 추가해줘" |
| UI/목업 | "화면 만들어줘", "목업 생성" |

## 예외 (자동 실행 안 함)

| 조건 | 처리 방식 |
|------|----------|
| 단순 질문/설명 요청 | 직접 응답 ("이게 뭐야?", "설명해줘", "어떻게 동작해?") |
| 파일 읽기/확인 | 직접 Read ("이 파일 보여줘", "현재 상태 확인") |
| `!quick`, `!just`, `!hotfix` | Magic Word → 즉시 실행, /auto 스킵 |
| `/`로 시작하는 명시적 스킬 | 해당 스킬 실행 (`/commit`, `/check`, `/debug` 등) |
| git 상태 확인 | 직접 처리 ("커밋 로그 보여줘", "브랜치 상태") |
| 대화 이어가기 | 진행 중인 /auto 워크플로우 내 후속 응답 |
| `--no-auto` 명시 | 사용자가 자동 실행을 명시적으로 거부 |

## 실행 방법

예외가 아닌 모든 작업 요청에 대해:

```
Skill("auto", args="{사용자의 원문 요청}")
```

사용자가 옵션을 포함한 경우 그대로 전달:
- "이미지 분석해줘" → `Skill("auto", args="이미지 분석해줘")`
- "목업 만들어줘 --figma" → `Skill("auto", args="목업 만들어줘 --figma")`

## 금지

- 작업 요청인데 "혹시 /auto 실행할까요?" 질문 금지 — 자동 실행
- 예외 조건에 해당하는데 /auto 강제 실행 금지
- /auto 실행 중에 중복 /auto 호출 금지
